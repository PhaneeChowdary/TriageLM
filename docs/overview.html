Portfolio project

Ticket Triage: fine-tuning a small LLM for structured output

Turn a raw customer support message into a strict JSON object (intent, category, entities, priority, routing) with a QLoRA-fine-tuned Llama-3.2-3B. A "form, not facts" task, done the disciplined way: measure first, fine-tune second, stay honest about the numbers.

Intent F1 0.585 to 1.000 OOD F1 0.344 to 0.876 Prompt tokens -86% 27 tests, lint clean

What this is (and is not)

The single decision the whole project hangs on.

Fine-tuning is good at teaching a model a form: a fixed output shape and a stable set of labels. It is bad at teaching a model facts, because facts change and a fine-tune freezes them in place.

In scope: triage

Classify the message into intent, category, entities, and a routing decision. Fixed schema, stable labels. This is a form problem, so fine-tuning fits.

Out of scope: answering

Actually resolving the ticket needs current knowledge. That is a retrieval/RAG problem, not fine-tuning. Deliberately excluded.

Tech stack

All free, reproducible, and split cleanly between local (CPU) and GPU.

Data and modelling

Hugging Face datasets and transformers, pydantic v2 for the schema, pandas, scikit-learn for the split and metrics.

Fine-tuning (GPU)

Unsloth + TRL + PEFT + bitsandbytes for QLoRA on a free Kaggle T4. Adapter and GGUF pushed to the Hugging Face Hub.

Inference and eval

Ollama runs both the base model and the fine-tuned GGUF locally, behind one generator interface, so evals need no GPU.

Tooling

uv for the environment and pinned deps, YAML-driven configs, pytest and ruff, a Makefile, and optional Weights and Biases tracking.

The output schema

Every ticket becomes exactly this object. Extra keys are rejected.

{
  "intent": "cancel_order",            // 1 of 27 gold labels
  "category": "ORDER",                 // 1 of 11 gold labels
  "entities": {                        // 8 fixed slots, null when absent
    "order_id": "{{Order Number}}",
    "account_type": null, "account_category": null, "person_name": null,
    "refund_amount": null, "delivery_city": null,
    "delivery_country": null, "invoice_id": null
  },
  "priority": "high",                  // derived by rules
  "requires_human": false,             // derived by rules
  "suggested_queue": "orders"          // derived by rules
}
Honesty note: priority, requires_human, and suggested_queue are not in the dataset. They come from a documented, versioned rules policy that I authored, so the model can only be as good as that policy. This is written down, not hidden.

Read the clean 1.000 with skepticism. Bitext is synthetic and templated, so the test set shares templates with training and the model essentially memorizes them. The OOD numbers are the honest signal; a real messy-data test is the true bar and is the planned next step.

How to run

Local half is CPU-only; only training needs a GPU.

# 1. Local pipeline + tests (no GPU)
make setup && make test && make smoke
make data

# 2. Baseline gate (needs Ollama)
ollama pull llama3.2:3b
make eval-baseline

# 3. Fine-tune on Kaggle
make bundle            # upload kaggle_bundle.zip + run notebooks/train_qlora.ipynb

# 4. Evaluate the fine-tune and compare
ollama create triage-ft -f Modelfile
make eval-finetuned
make compare

Why this project, not another chatbot

Right tool for the task

Triage is a form problem, so fine-tuning is the correct choice, and the README says exactly when it is not.

Eval before model

The scoring harness was built and smoke-tested before any GPU spend. A fine-tune you cannot measure is just vibes.

A real gate

The base model was measured first (intent F1 0.585). Fine-tuning only proceeded because there was clear room to improve.

Intellectual honesty

Authored labels are flagged as policy, and the too-perfect clean score is called out rather than celebrated.
